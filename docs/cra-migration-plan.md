# CRA Migration Plan

This project still runs on `react-scripts` (Create React App) even though the
frontend now uses React `19.x`, TypeScript `4.9`, managed Azure Static Web Apps
APIs, and a more feature-grouped codebase than CRA was really designed for.

The app still builds, but the current setup is a toolchain smell:

- CRA is effectively legacy infrastructure now.
- Build output and dependency behavior are harder to reason about than they need
  to be.
- React 19 support works here, but not because CRA is a healthy long-term fit.
- Debugging and future build customization will keep getting more awkward.

The recommendation is to migrate the frontend from CRA to `Vite`.

## Current constraints

Any migration has to preserve these current realities:

- Static hosting target is Azure Static Web Apps
- The managed API lives in `/api`
- The frontend deploy workflow builds the app twice:
  - `REACT_APP_CONTENT_PACK=public`
  - `REACT_APP_CONTENT_PACK=team`
- SPA fallback is defined in
  [`fredagskoll-frontend/public/staticwebapp.config.json`](../fredagskoll-frontend/public/staticwebapp.config.json)
- Build metadata is generated by
  [`fredagskoll-frontend/scripts/write-build-info.js`](../fredagskoll-frontend/scripts/write-build-info.js)
- Public assets and app icons are still expected in the current public-folder
  shape

## Recommended target

Use:

- `Vite`
- `React`
- `TypeScript`

Keep:

- Azure Static Web Apps
- managed `/api`
- the current repo layout
- the dual public/team deployment model

## Migration phases

### 1. Prepare the frontend for toolchain swapping

Goal: reduce CRA-specific assumptions before any build swap.

Tasks:

- audit direct usage of `process.env.REACT_APP_*`
- introduce a tiny env accessor module for frontend config
- confirm every static asset path works without CRA-only magic
- keep `buildInfo.generated.ts` generation independent from the bundler

Done when:

- frontend config is read through one place
- asset assumptions are explicit
- no React code depends directly on CRA internals

### 2. Scaffold Vite in-place

Goal: prove the app can build under Vite without changing product behavior.

Tasks:

- add `vite.config.ts`
- add Vite React plugin
- keep the existing `src/` structure
- switch the HTML entry to Vite format
- preserve `public/` asset behavior

Done when:

- `npm run dev` works under Vite
- `npm run build` produces `dist/`
- the app renders correctly locally

### 3. Replace CRA env handling

Goal: remove `REACT_APP_*` dependency.

Tasks:

- replace frontend env reads with `import.meta.env`
- rename public/team pack variable to something Vite-friendly
  - recommended: `VITE_CONTENT_PACK`
- verify local startup and Azure build env injection for both app variants

Done when:

- no frontend code depends on `process.env.REACT_APP_*`
- public/team builds still separate correctly

### 4. Update Azure deployment

Goal: make GitHub Actions deploy the Vite build output cleanly.

Tasks:

- update `.github/workflows/deploy-static-apps.yml`
- change `output_location` from `build` to `dist`
- verify both Static Web Apps still deploy with `api_location: api`
- confirm SPA fallback still works with the existing
  `staticwebapp.config.json`

Done when:

- both public and team apps build and deploy from Vite
- managed API behavior is unchanged

### 5. Remove CRA

Goal: finish the migration cleanly.

Tasks:

- remove `react-scripts`
- remove CRA-specific scripts and assumptions
- clean up TypeScript and ESLint configuration if needed
- refresh README and toolchain docs

Done when:

- the repo no longer depends on CRA
- local dev, tests, and Azure deploy all run through the new toolchain

## Known risk areas

### Environment variables

This is the most obvious migration edge.

Today the app relies on:

- `REACT_APP_CONTENT_PACK`

That needs a controlled swap to:

- `VITE_CONTENT_PACK`

The safest path is to add a small config helper first, then switch the helper
implementation during migration.

### Static Web Apps deployment shape

The current Azure workflow expects CRA output in `build/`.

Vite will output `dist/`, so the workflow must be updated in lockstep with the
build migration.

### Asset paths

The app uses public assets, generated branding, GIFs, and image metadata. Those
paths need to be checked carefully under Vite so the migration does not become a
death-by-a-thousand-broken-files exercise.

## Suggested order of execution

1. add a frontend config helper
2. scaffold Vite without removing CRA yet
3. make the app run locally under Vite
4. swap env handling
5. update Azure workflow
6. test both public/team deploys
7. remove CRA

## Not in scope

- changing the managed API model
- replacing Azure Static Web Apps
- redesigning the frontend
- changing the public/team product split

The goal is toolchain cleanup, not product churn.
